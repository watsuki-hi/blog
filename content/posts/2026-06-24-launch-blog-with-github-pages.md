今日は、この情シスブログを立ち上げました。

やったことは、Markdownで記事を書けるようにして、HTMLを生成して、GitHub Pagesで公開するところまでです。

ブログを作るだけなら、WordPress、はてなブログ、note、Zenn、Qiitaなど、選択肢はいろいろあります。最初から読者がいる場所に書くなら、そういうサービスを使うのも良い選択です。

ただ今回は、個人の情シスノートとして、自分で構成を決められて、運用も軽く、セキュリティ面でも余計な入口を持たない形にしたかったので、静的サイトとして作ることにしました。

## 最初に決めたこと

最初に決めたのは、できるだけ小さく始めることです。

ブログを始めるときは、デザイン、CMS、独自ドメイン、アクセス解析、問い合わせフォーム、カテゴリ設計、SEOなど、考えようと思えばいくらでも考えることがあります。

でも、最初から全部を決めようとすると、肝心の記事を書くところまでたどり着きません。

今回の目的は、きれいなブログシステムを作ることではなく、情シスの仕事で得た知見を残せる場所を作ることです。

なので、最初の条件はかなり絞りました。

- Markdownで記事を書けること
- Gitで履歴を残せること
- 無料または低コストで公開できること
- サーバーやデータベースを持たないこと
- ログイン画面や管理画面を公開しないこと
- 記事を追加したら自動で公開できること

この条件で考えると、静的サイトとGitHub Pagesの組み合わせがちょうどよさそうでした。

## なぜ静的サイトにしたか

静的サイトは、完成したHTML、CSS、画像だけを公開する構成です。

WordPressのように、アクセスのたびにデータベースから記事を取り出してページを組み立てるわけではありません。

記事を書くときはMarkdownで書きます。公開するときにHTMLへ変換します。公開先には、変換済みのファイルだけを置きます。

この構成の良いところは、動く部品が少ないことです。

サーバーのPHPバージョン、データベース、管理画面、プラグイン、テーマ更新などを気にしなくて済みます。もちろんGitHub Pages側の仕組みには依存しますが、自分で管理する対象はかなり少なくなります。

情シスの視点で見ると、「管理するものを増やさない」はかなり大事です。

個人ブログのためにサーバーを持つと、パッチ適用、バックアップ、監視、証明書、ログ管理、不要なアカウントの整理など、ブログを書くこと以外の運用が増えます。

それが楽しい場合もありますが、今回はブログ本文を残すことが目的です。

だから、できるだけ運用負荷が小さい構成にしました。

## 作ったディレクトリ構成

今回のブログは、かなり単純な構成です。

- content/posts に記事のMarkdownを置く
- src/build.js でHTMLを生成する
- src/styles.css で見た目を調整する
- public に画像などの静的ファイルを置く
- dist に公開用の完成ファイルを出力する
- .github/workflows/pages.yml でGitHub Pagesへ公開する

記事を書くときに触るのは、基本的には content/posts です。

デザインを直すときは src/styles.css を触ります。

HTMLの構造やトップページの並びを変えたいときは src/build.js を触ります。

dist はビルド結果なので、基本的には直接編集しません。MarkdownやCSSを直してから、npm run build で作り直します。

この分け方にしておくと、「原稿」「生成処理」「見た目」「公開物」が混ざりにくくなります。

## Markdown記事の形

記事はMarkdownファイルとして作ります。

先頭には、タイトル、日付、概要、タグを書きます。

たとえば、先頭に次のような情報を書きます。

- title: 記事タイトル
- date: 2026-06-24
- excerpt: 一覧に表示する短い説明
- tags: 情シス、セキュリティ

この下に本文を書きます。

このメタ情報をもとに、トップページの記事一覧や、記事ページのタイトル、日付、タグを生成します。

この形にしておくと、記事本文と管理情報がひとつのファイルにまとまります。あとから見返したときにも分かりやすいです。

## HTMLを生成する仕組み

HTML生成は、src/build.js というNode.jsのスクリプトで行っています。

やっていることは大きく分けると、次のような流れです。

- content/posts のMarkdownファイルを読む
- 先頭のメタ情報を取り出す
- Markdown本文をHTMLに変換する
- トップページの記事一覧を作る
- 各記事ページを作る
- public の画像などを dist にコピーする
- styles.css を dist にコピーする
- sitemap.txt を作る

今のMarkdown変換はかなりシンプルです。

対応しているのは、見出し、段落、箇条書き、太字、リンク、インラインコードくらいです。

本格的なMarkdownパーサーを使えば、コードブロック、表、引用、脚注などにもきれいに対応できます。

ただ、最初から大きな仕組みにすると、理解するものも増えます。

今回はまず、「記事を書いて公開できる」ことを優先しました。必要になったら、あとからMarkdownライブラリを入れればよいと考えています。

## GitHub Pagesで公開する

公開先にはGitHub Pagesを使いました。
GitHub Pagesは、GitHubリポジトリの内容をWebサイトとして公開できる仕組みです。

今回はGitHub Actionsでビルドして、dist の中身をPagesへデプロイする形にしました。

流れとしては、mainブランチにpushすると、GitHub Actionsが動きます。

Actionsでは、Node.jsを準備して、npm run build を実行します。
ビルドで dist が作られたら、それをGitHub Pagesへアップロードして公開します。

つまり、ローカルで記事を書いて、コミットして、pushすれば、自動で公開まで進みます。

これはかなり楽です。

手元でFTPアップロードをする必要もありません。公開用ファイルだけを手作業でコピーする必要もありません。

Gitの履歴を見れば、いつどの記事を追加したか、どんなデザイン変更をしたかも分かります。

## つまずいたところ

今回つまずいたところもありました。

ひとつは、リポジトリ名を変えたことです。

最初は watsuki-blog という名前で作っていましたが、途中で blog に変えました。
GitHub Pagesで project site として公開する場合、URLにはリポジトリ名が入ります。

つまり、リポジトリ名が blog なら、公開URLは次のようになります。

https://watsuki-hi.github.io/blog/

このとき、サイト内のCSSや記事リンクも /blog から始まるようにしておく必要があります。

そこで、GitHub Actionsのビルド時に BASE_PATH=/blog を指定しました。

これを古い /watsuki-blog のままにしていると、ページは表示されてもCSSやリンクがずれる可能性があります。

もうひとつは、リポジトリの公開設定です。

途中でリポジトリをprivateにしたら、GitHub Pagesが404になりました。

GitHub Pagesは、プランやリポジトリの公開状態によって使える条件が変わります。

無料プランで普通に公開するなら、publicリポジトリにするのが分かりやすいです。

このあたりは、エラー画面だけを見ると原因が分かりにくいです。

「Pagesの設定が悪いのか」「Actionsが失敗しているのか」「URLが違うのか」「privateにしたせいなのか」を切り分ける必要がありました。

今回は、Actionsは成功しているのにPagesのURLが404になっていたので、公開設定とリポジトリの可視性を確認しました。

## デザインを直した

最初に作ったデザインは、大きなヒーロー画像があり、色も少し強めでした。

見た目としては整っていましたが、少しAIっぽく感じました。

このブログで主役にしたいのは、画像や派手なトップページではなく記事です。

なので、デザインをかなりシンプルに直しました。

具体的には、大きな画像ヒーローをやめました。

色数も減らしました。

白、黒、グレーを中心にして、アクセントカラーはリンクやホバーくらいにしました。

記事カードも、影のあるカード型ではなく、日付とタイトルが並ぶ一覧型にしました。

トップページを開いたときに、なるべく早く記事一覧が見えるようにしています。

フォントも調整しました。

最初はInterやNoto Sans JPを使っていましたが、少し今っぽすぎる印象がありました。

そこで、昔ながらの日本語ブログに近い、Helvetica Neue、Helvetica、Arial、ヒラギノ角ゴ、メイリオ、MS PGothicの指定にしました。

細かいところですが、フォントだけでもブログの空気はかなり変わります。

## 今の運用手順

今後の記事追加は、だいたい次の流れになります。

まず content/posts にMarkdownファイルを作ります。

次に本文を書きます。

ローカルで確認するときは、npm run build と npm run serve を使います。

問題なければ、git add、git commit、git push をします。

pushするとGitHub Actionsが動いて、GitHub Pagesに公開されます。

手順としては単純です。

単純ですが、単純だからこそ続けやすいと思っています。

## 情シスブログとして残したいこと

このブログでは、情シスの仕事で得た知見を残していきます。

セキュリティ、SaaS運用、アカウント管理、PC運用、問い合わせ対応、社内ルール作りなど、現場では毎日のように小さな判断があります。

その判断は、派手なノウハウではないかもしれません。

でも、「なぜその設定にしたのか」「なぜその運用にしたのか」「どこでつまずいたのか」を残しておくと、あとから役に立ちます。

情シスの仕事は、やったことが見えにくいことも多いです。

何も起きない状態を作るために、裏側で細かい整備を続ける仕事でもあります。

だからこそ、考え方や手順を文章にしておく意味があります。

自分のための記録でもあり、同じように小さな現場で運用している人に届けばよい、という気持ちです。

## 今日できたこと

今日できたことをまとめると、次の通りです。

- ブログ用のディレクトリを作った
- Markdown記事を置く場所を作った
- HTMLを生成するスクリプトを作った
- CSSで見た目を整えた
- GitHubリポジトリを用意した
- GitHub Pagesで公開した
- リポジトリ名変更後のBASE_PATHを直した
- privateリポジトリによる404を切り分けた
- 記事が主役になるようにデザインを直した
- フォントを昔ながらのブログ寄りに調整した

こうして見ると、ブログを立ち上げるだけでも、いろいろな小さな判断があります。

公開先、URL、ビルド方法、デザイン、フォント、記事の書き方、Gitの運用。

どれも単体では小さいですが、積み上げると「続けられる場所」になります。

## 今後やりたいこと

まずは記事を増やしていきます。

最初からカテゴリやタグを細かく作り込むよりも、実際に書いた記事が増えてから整理した方がよさそうです。

Markdownの表現力も、必要になったら広げます。

コードブロック、表、引用、目次などは、そのうち欲しくなるかもしれません。

でも今は、文章を書ける状態を大切にします。

ブログは、仕組みを作って終わりではありません。

記事を書いて、見直して、必要なら直して、少しずつ育てていくものです。

今日作ったのは、そのための最初の土台です。
